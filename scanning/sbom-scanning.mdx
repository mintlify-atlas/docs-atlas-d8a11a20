Grype can scan a Software Bill of Materials (SBOM) directly instead of pulling and cataloguing a container image or filesystem at scan time. This approach is faster and does not require Docker or any container runtime.

## Benefits of SBOM scanning

- **No container runtime required.** You can scan from a CI environment without Docker or Podman.
- **Faster scans.** Grype skips the cataloguing step and goes straight to vulnerability matching.
- **Works with any SBOM producer.** Grype accepts SBOMs generated by Syft, Trivy, cdxgen, or any other tool that produces a supported format.
- **Reusable artifact.** Generate the SBOM once and scan it multiple times or archive it for later use.

## Supported SBOM formats

Grype accepts the following SBOM formats as input:

| Format | Notes |
|--------|-------|
| Syft JSON | Native Grype format with the richest package metadata |
| CycloneDX XML | CycloneDX v1.2–v1.6 |
| CycloneDX JSON | CycloneDX v1.2–v1.6 |
| SPDX Tag-Value | SPDX 2.x |
| SPDX JSON | SPDX 2.x |

<Note>
  Syft JSON provides the most complete package metadata (CPEs, PURLs, file locations) for Grype to match against. When generating SBOMs for vulnerability scanning, Syft JSON is recommended.
</Note>

## Scanning an SBOM file

Use the `sbom:` scheme to scan an SBOM from disk:

```bash
grype sbom:./sbom.json
```

You can use an absolute path too:

```bash
grype sbom:/path/to/sbom.cdx.json
```

## Generating an SBOM with Syft, then scanning

<Steps>
  <Step title="Install Syft">
    Install the Syft SBOM generator:

    ```bash
    curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin
    ```
  </Step>
  <Step title="Generate an SBOM">
    Generate a Syft JSON SBOM for an image:

    ```bash
    syft alpine:latest -o json > sbom.json
    ```

    Or for a directory:

    ```bash
    syft dir:./my-project -o json > sbom.json
    ```
  </Step>
  <Step title="Scan the SBOM">
    Pass the saved SBOM to Grype:

    ```bash
    grype sbom:./sbom.json
    ```
  </Step>
</Steps>

## Piping an SBOM directly

You can pipe Syft output directly into Grype without writing an intermediate file:

```bash
syft alpine:latest -o json | grype
```

Grype reads from stdin when no argument is given and stdin is a pipe or redirect. This works with any SBOM format Grype supports:

```bash
# Pipe CycloneDX JSON
syft alpine:latest -o cyclonedx-json | grype

# Pipe SPDX JSON
syft alpine:latest -o spdx-json | grype
```

## Generating CPEs for packages missing CPE data

Some SBOMs — particularly those generated by third-party tools or from SPDX documents — may not include CPE (Common Platform Enumeration) data for every package. Without CPEs, Grype may miss some vulnerability matches that rely on CPE-based lookup.

Use `--add-cpes-if-none` to instruct Grype to generate CPEs for any package that has none:

```bash
grype sbom:./sbom.spdx.json --add-cpes-if-none
```

<Tip>
  `--add-cpes-if-none` is especially useful when scanning SBOMs produced by tools other than Syft, which may omit CPE data.
</Tip>

## Specifying the distro

When scanning an SBOM that contains OS packages but lacks distro information, Grype may not be able to resolve the correct vulnerability feed. Use `--distro` to supply the distro hint manually:

```bash
grype sbom:./sbom.json --distro alpine:3.20
grype sbom:./sbom.json --distro ubuntu:22.04
```

The format is `<distro>:<version>` or `<distro>-<version>`.
